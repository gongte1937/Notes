# Database Choices 总结

## 一、选择依据

数据库选型主要取决于：

- 数据结构类型
- 访问模式（读多 / 写多）
- 扩展需求（垂直扩展 / 横向扩展）
- 一致性要求（强一致 / 最终一致）
- 延迟与吞吐量目标

---

## 二、数据库类型

### 1. Relational Database（SQL）

示例：PostgreSQL、MySQL

特点：

- 结构化数据（表 + Schema）
- 强一致性（ACID）
- 支持 JOIN 与复杂查询

适用场景：金融系统、订单系统、用户账户、事务型业务（OLTP）

优缺点：

- 优点：强一致、数据完整性高、成熟稳定
- 缺点：横向扩展困难，JOIN 在大规模系统中成本高

---

### 2. NoSQL — Key-Value Store

示例：Redis、DynamoDB

特点：

- Key → Value 存储，访问极快
- 易于横向扩展
- 通常不支持复杂查询

适用场景：缓存、Session、简单状态存储、高频读写

优缺点：

- 优点：高性能、简单、易扩展
- 缺点：查询能力弱，不适合复杂关系数据

---

### 3. NoSQL — Document Store

示例：MongoDB

特点：

- 文档模型（JSON/BSON）
- 支持嵌套数据，减少 JOIN

适用场景：CMS、用户画像、产品目录、半结构化数据

优缺点：

- 优点：灵活、适合快速迭代、读性能好
- 缺点：强一致性弱于传统 SQL，复杂事务能力有限

---

### 4. NoSQL — Wide Column Store

示例：Cassandra、Google Bigtable

特点：

- 列式存储，高写入吞吐
- 面向大规模分布式系统，强扩展能力

适用场景：日志系统、物联网数据、时间序列、写多读少

优缺点：

- 优点：横向扩展能力强、高可用、写入性能高
- 缺点：查询模式受限，不适合复杂 JOIN

---

### 5. Search Engine

示例：Elasticsearch

特点：

- 倒排索引、全文搜索
- 支持复杂过滤与聚合

适用场景：站内搜索、日志分析、商品搜索、内容检索

优缺点：

- 优点：搜索能力强、分布式扩展、支持模糊匹配
- 缺点：不适合事务型写入，数据一致性相对较弱

---

### 6. Blob / Object Storage

示例：Amazon S3

特点：

- 存储大文件，高耐久性、低成本、可扩展

适用场景：图片、视频、日志归档、备份文件

优缺点：

- 优点：高可靠、便宜、几乎无限扩展
- 缺点：查询能力弱，不适合高频事务操作

---

## 三、如何选择数据库（面试思路）

可以从以下角度回答：

### 1. 数据结构

- 强结构化 → SQL
- 半结构化 → Document
- 简单 KV → Key-Value

### 2. 访问模式

- 高频读 → Cache / Key-Value
- 高频写 → Wide Column
- 搜索需求 → Search Engine

### 3. 一致性要求

- 强一致 → SQL
- 最终一致 → NoSQL

### 4. 扩展能力

- 垂直扩展 → SQL
- 横向扩展 → NoSQL / Wide Column

---

## 四、一句话总结

> 不同数据库针对不同工作负载，按数据结构、访问模式、一致性与扩展性做权衡选择。

| 类型 | 核心优势 | 典型场景 |
| --- | --- | --- |
| SQL | 强一致 + 事务 | 订单 / 金融 |
| Key-Value | 极快读写 | 缓存 / Session |
| Document | 灵活结构 | 用户数据 |
| Wide Column | 高写吞吐 | 日志 / 时间序列 |
| Search | 强搜索能力 | 商品/内容检索 |
| Object Storage | 大文件存储 | 图片 / 视频 |
