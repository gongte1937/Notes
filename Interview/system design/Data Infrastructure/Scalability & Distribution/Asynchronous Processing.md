# Asynchronous Processing 总结

## 一、异步处理是什么？

Asynchronous Processing（异步处理）是把耗时任务从主请求链路中拆出去，交给后台 Worker 执行。

核心思想：

> 主流程先快速响应，重任务后置处理。

---

## 二、为什么要做异步？

同步链路通常是：

```text
User -> API -> 执行耗时任务 -> 返回
```

问题在于：

- 响应时间长，用户等待明显
- 请求线程被占用，吞吐受限
- 上游压力放大，容易级联超时
- 高峰期更容易触发雪崩

---

## 三、典型架构模型

```text
User
  ↓
API Server (Producer)
  ↓
Message Queue
  ↓
Worker Pool (Consumer)
```

基本流程：

1. API 接收请求并校验参数
2. 任务写入 Queue（持久化）
3. API 立即返回（例如 202 Accepted）
4. Worker 消费任务并执行
5. 执行结果写回 DB / Cache / 回调通知

---

## 四、常见应用场景

- 邮件、短信、推送通知
- 报表生成与导出
- 图片缩略图、视频转码
- 搜索索引更新
- 数据同步与事件分发
- 日志聚合与离线计算

---

## 五、核心收益

- 降低用户感知延迟（p99 响应时间更可控）
- 解耦服务边界（生产者与消费者独立演进）
- 提升系统吞吐（Worker 可水平扩展）
- 削峰填谷（Queue 做缓冲层）
- 提升整体稳定性（慢任务不阻塞主链路）

---

## 六、常见通信模式

### 1. Producer-Consumer

最常见模式：一个生产者写任务，多个消费者并行处理。

### 2. Queue（点对点）

一条消息通常只被一个消费者处理，适合任务型场景。

常见组件：

- RabbitMQ
- Kafka
- Redis Streams
- AWS SQS

### 3. Pub/Sub（发布订阅）

一个事件可被多个下游消费，适合事件驱动和服务解耦。

---

## 七、可靠性设计（面试高频）

### 1. 消息不丢

- Broker 持久化
- Producer Confirm / ACK
- Consumer ACK 后再提交偏移

### 2. 消费语义

大多数系统默认是 `at-least-once`，因此要接受“可能重复消费”。

### 3. 幂等性

- 任务唯一 ID（幂等键）
- 数据库唯一约束 / 去重表
- 状态机防重（只允许合法状态迁移）

### 4. 失败重试

- 指数退避（Exponential Backoff）
- 最大重试次数
- Dead Letter Queue（DLQ）兜底

### 5. 顺序保障

需要顺序时按业务 key 分区（例如 `user_id`），同 key 保证单分区有序消费。

---

## 八、与同步处理对比

| 对比项 | 同步 | 异步 |
| --- | --- | --- |
| 首次响应延迟 | 高 | 低 |
| 链路耦合 | 高 | 低 |
| 峰值承载 | 较弱 | 较强 |
| 实现复杂度 | 低 | 高 |
| 一致性模型 | 更偏强一致 | 常见最终一致 |

---

## 九、常见风险与治理

- 消息堆积：加消费者、限流、扩分区、降级非核心任务
- 重复消费：幂等设计必须前置
- 顺序错乱：按 key 路由并限制并发模型
- 可观测性不足：补齐队列深度、消费延迟、失败率监控
- 毒消息阻塞：快速转移到 DLQ 并告警

---

## 十、与可扩展架构的关系

异步处理属于 `Scalability & Distribution` 的关键能力，和 [[Load Balancing]] 配合时效果更明显：

- Load Balancing 解决入口分发
- Asynchronous Processing 解决慢任务后置

两者结合可同时优化“实时响应”和“系统吞吐”。

---

## 十一、面试回答框架

1. 先说明哪些操作必须同步，哪些可以异步
2. 设计 Producer -> Queue -> Worker 主链路
3. 明确消息语义（至少一次）与幂等方案
4. 设计重试、退避、DLQ 与告警
5. 说明扩展方式（分区 + Worker 水平扩容）
6. 补充监控指标（堆积深度、消费延迟、失败率）

---

## 十二、一句话总结

> 异步处理的本质是把“慢且可延后”的任务从主链路剥离，用队列和 Worker 换取响应速度、吞吐和稳定性。
