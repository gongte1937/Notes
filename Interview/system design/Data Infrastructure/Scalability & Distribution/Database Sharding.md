# Database Sharding 总结

## 一、Sharding 是什么？

Sharding（分片）是：

> 把一张大表的数据拆分到多台数据库服务器上存储。

本质是：

> 用多台机器分担一张表的数据和压力。

属于 **横向扩展（Horizontal Scaling）**。

---

## 二、为什么需要 Sharding？

当出现：

- 数据量过大（单机存不下）
- 单机 CPU / 内存 / 磁盘达到瓶颈
- 高并发访问导致性能下降

单机 + Index 已经无法解决问题。这时需要：

> 把数据拆开，分布到多个数据库实例。

---

## 三、Sharding 的基本原理

假设表：

```sql
users(id, name, subbed)
```

原来：

```text
DB Server
└── users（全部数据）
```

Sharding 后：

```text
DB1 → 部分用户
DB2 → 部分用户
DB3 → 部分用户
DB4 → 部分用户
```

每个数据库内部仍然有：

- Heap File
- Index
- Block

只是数据被拆开了。

---

## 四、常见分片策略

### 1. 按 ID 取模（最常见）

`user_id % 4`

| user_id | shard |
| --- | --- |
| 1 | DB1 |
| 2 | DB2 |
| 3 | DB3 |
| 4 | DB4 |
| 5 | DB1 |

优点：

- 实现简单
- 数据分布均匀

缺点：

- 扩容困难（改 shard 数量会导致大量数据迁移）

---

### 2. 按范围分片

1 - 100万 → DB1  
100万 - 200万 → DB2

优点：

- 易于理解
- 范围查询友好

缺点：

- 容易出现热点（高 ID 区间可能集中在一个 shard）

---

### 3. 按业务维度分片

例如：

中国用户 → DB1  
美国用户 → DB2  
欧洲用户 → DB3

适用于：

- 强业务隔离场景

---

## 五、查询流程（Sharding 后）

查询：

```sql
SELECT * FROM users WHERE id = 6;
```

流程：

1. 计算分片规则（如 `6 % 4`）
2. 定位到对应 shard（DB2）
3. 在该 shard 内部用 index 查询

关键点：

> 只访问一个 shard，而不是所有数据库。

---

## 六、Sharding 的优点

- 提高系统整体吞吐量
- 单机压力下降
- 支持海量数据
- 可横向扩展（加机器）

---

## 七、Sharding 的问题

### 1. 跨分片查询复杂

```sql
SELECT COUNT(*) FROM users;
```

需要：

- 所有 shard 分别计算
- 汇总结果

---

### 2. 事务变复杂

跨 shard 事务成本高。

---

### 3. 扩容成本高

增加 shard 数量：

- 需要数据迁移
- 可能影响线上业务

---

### 4. 热点问题

某些 shard 可能流量特别高。

---

## 八、Sharding vs Index

| 概念 | 解决问题 |
| --- | --- |
| Index | 单机内如何更快查找 |
| Sharding | 单机放不下怎么办 |

---

## 九、核心理解

Sharding 不是加速查询的工具。

它是：

> 当单机撑不住时，用多台机器分担数据和流量的架构方案。

---

## 十、一句话总结

> 把一张大表拆到多台数据库上，用横向扩展解决容量和性能瓶颈。
